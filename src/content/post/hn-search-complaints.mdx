We've [launched a new Hacker News search experience](https://trieve.ai/launching-trieve-hn-discovery/), focused on discovery: [hn.trieve.ai](https://hn.trieve.ai/) (GitHub: [Trieve API backend](https://github.com/devflowinc/trieve), [frontend search interface](https://github.com/devflowinc/trieve-hn-discovery/)).

Hacker News has long been a playground for search innovation—with the community often leaning in to explore new possibilities in search. Over the past six months, Nick has been looking back at the various search experiences and detailed his findings in a post: [History of HackerNews Search: From 2007 to 2024](https://trieve.ai/history-of-hnsearch/).

We combed through HN (and [user issues posted to Algolia's repo for HN search](https://github.com/algolia/hn-search/issues)) in search of search complaints. Over the years there have been some complaints about indexing issues, and we’re not covering those in this post. Instead, we looked for examples where people shared actual search queries. For each query, we looked for what they said or implied about their search intent and the search results they found. What have people said about search quality? What searches are not possible or not easy in the current HN search? When are folks resorting to running a `site:` search on a search engine like Google? Where can Trieve help make search better?

## Discover well-beyond exact matches in titles

### Searching for "postgres clustering"

- **Search intent:** Finding information about PostgreSQL clustering solutions
- **Sourced from:** [@avereveard comment on Hacker News](https://news.ycombinator.com/item?id=40240237)

Algolia's engine focuses on showing extremely precise keyword matches while Trieve's takes a more discovery-focused approach and prioritizes relevance. This query is a good example of where a relevance focused approach can be better. However, it's important to note that there are pros and cons to each approach.

## Searching with special characters

### Searching for "[video]"

- **Search intent:** Exact string match when string contains non-alphanumeric characters
- **Sourced from:** [@some1else on github/algolia/hn-search/issues](https://github.com/algolia/hn-search/issues/197)

Algolia&#x27;s result appear to match on <code>video</code>, not only <code>[video]</code>. Trieve&#x27;s hybrid and semantic search types do very well on the exact match, but not perfect. Trieve does better if in quotes, like [&quot;[video]&quot;], which does not seem to have an affect here in Algolia. Difference here is likely due to tokenization approaches moreso than general relevance strategy. Again, there are pros and cons to different tokenization strategies, but issues similar to this one helped inform our final decision on approach.

### Searching for "AT&amp;T"

- **Search intent:** Exact match for acronym (with ampersand)
- **Sourced from:** [@Pranoy1c on github/algolia/hn-search/issues](https://github.com/algolia/hn-search/issues/244)

The issue seems to be with the prefix paramater in Algolia. Algolia&#x27;s results are less relevant when searching for [AT&amp;T] with <code>prefix=true</code>. Results improve with <code>prefix=false</code> or when the query is in quotes. This appears to be an issue with how the <code>prefix</code> parameter is set in the URL in Algolia. If the user opens the Algolia HN search and starts typing <code>prefix=true</code>, if the user hits enter, the URL is changed to <code>prefix=false</code>. Trieve&#x27;s results (for all search types) are relevant even without quotes. There may be reasoning for Algolia's default prefixing params outside of this query which make it better overall, but we indexed on this complaint.

## Out-of-domain strings

### Searching for "lootitooti"

- **Search intent:** Match for out-of-domain string
- **Sourced from:** [@douglaskayama comment on Hacker News](https://news.ycombinator.com/item?id=40869786)

This was actually a complaint about not Algolia, but a demo search tool from another provider. The user praised Algolia&#x27;s performance on this query. Trieve&#x27;s different search types support different user goals, so while BM25 works great for out-of-domain strings like this, the others would require quotation marks. Queries like this one tend to favor Algolia's more narrow and precise strategy over Trieve's relevance focus. Our engine splits "lootitooti" into multiple tokens like "loot", "lootit", etc. instead of preserving the entire match. Using Trieve's internal BKTree to detect non-dictionary tokens and auto-quote require them is the clearest fix for these kinds of queries, but we have not yet implemented it in the HN search demo.
